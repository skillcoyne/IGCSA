Introduction

FIGG is a genome simulation tool that uses known or theorized variation probability, per a given fragment size and grouped by GC content across a genome to generate new genomes in FASTA format while tracking applied mutations for use in analysis tools or population simulations.


Installation

Java 1.6 or higher is required to run FIGG. Download and expand one of the provided compressed packages from the SF download page. This package includes a database and one FASTA file for an example run.

Run Provided Example

After uncompressing the package in a Unix or OSX system open a terminal and run: 
sh figg.sh -n MyGenome -o -c 22
on Windows run cmd.exe and in the window run: 
figg.bat -n MyGenome -o -c 22
Chromosome 22 takes about 26 minutes to fragment and mutate on current hardware. A

In order to run a full genome

  • Download human FASTA files from NCBI or UCSC. Note that only chromosomes 1-22, X and Y are represented in the database.
  • Update the dir.assembly property in the genome.properties file to point to the directory where the FASTA files were saved
  • Alter the dir.insilico directory if you wish to save the mutated genomes in a new location.
  • Run the correct script for your OS (bat/sh file) without the -c option: 
    sh figg.sh -n MyGenome -o -t 12
Note: The full genome run using the thread option -t 12 takes 32-36 hours to complete on current hardware.

FAQ

Does FIGG work with all genomes?
Currently this has only been tested on human GRCh37. However, the tool itself should work with any genomic data in FASTA format. The database tables will need to be generated based on your specific genome and known/hypothesized variation information and the scripts available in the source download will require alteration to work for your files or new ones written.
What data is provided?
The data provided in the database tables were based on analysis of 1000Genomes and HapMap variations called against human reference GRCh37 (hg19). These variation tables can be used with any human genome sequence, however variation frequency may change between releases.
Can I include specific variations at known locations?
While the code is set up to support this, the current implementation does not provide for location based, or specific genomic alterations. This may be added in the future.

Usage

Usage: figg.sh (or figg.bat) [options]
    -n, --name Genome directory name, if not provided a random name is generated.
    -o, --overwrite Overwrite genome directory if name already exists.
    -c, --chromosome List of chromosomes to use/mutate, comma-separated (e.g. 21,3,X). If not included chromosomes will be determined by
    the fasta files found in the dir.assembly directory (see genome.properties).
    -t, --threads Number of concurrent threads, default is 5. Each thread handles a single chromosome.
    -h, --help Print usage help.
Examples using the provided shell script:
Generate mutations for chromosome 22, overwrite previous named run. 
figg.sh -n MyChr17 -o -c 22
Generate mutations for all chromosomes in the dir.assembly directory using 12 threads. Don't overwrite. 
figg.sh -n MyGenome -t 12

Notes & Warnings

Database Use
The Derby database is provided within the zip package. The provided setup connects directly (see genome.properties). If you intend to use this on a cluster it is currently recommended that you set up your cluster run to copy the derby database directory (db) to the local node for IO speed and update the properties file appropriately. If you prefer MySQL please download the database dump and alter the properties file appropriately. Note that no guarantees are made for speed if you choose to connect to a remote MySQL database from a cluster environment. While a local MySQL database is as fast as Derby, tests of the remote database show slowdowns of more than 50%.
